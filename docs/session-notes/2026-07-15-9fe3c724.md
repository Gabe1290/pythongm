## What I found before planning

Views/cameras are **already implemented on the desktop pygame runtime** — a genuinely complete GameMaker-style system: 8 views per room, border-based camera-follow with per-axis speed limiting, clamping to room bounds (`GameRunner.update_views()`, `runtime/game_runner.py:1247-1631`). `enable_views`/`set_view` action handlers exist and work.

But it's marked **"incomplete"** in `TODO.md` for two concrete reasons I confirmed:
1. **No IDE UI exposure** — `enable_views`/`set_view` have no entry in `events/action_types.py`, so there's no action-editor dialog or Blockly block for them. Usable today only by hand-authoring the JSON action dict or driving it from `execute_code` — exactly the pattern match3 already established.
2. **Zero export support** — `grep -rl "views_enabled\|view_offset" export/` returns **nothing**. Neither the Kivy exporter nor the Web/HTML5 exporter has any concept of a camera. A views-based game would export to Kivy/Web and either ignore the camera (render everything at absolute room coordinates, breaking the scroll) or worse.

That second gap is the same shape as the sound-queue/sprite-fallback gaps I closed for match3_2 this session — except bigger: sound was an *additive* queue bolted onto existing draw/sound paths. Views affect the **rendering transform of every single draw call**, so Kivy/Web support means threading a view offset through each exporter's whole render path, not adding one primitive.

**Given 68% of this session is already used, I'm proposing this as a plan to execute in a future session, not now.** Here's the shape I'd suggest:

**Phase 1 — engine work (do first, it's the actual risk):**
- Add view-offset-aware rendering to the Kivy exporter (base_object.py's render + draw-queue, scene camera transform) and to `engine.js` (canvas translate before drawing instances/tiles)

## [2026-07-17] User

> Gamemaker 6.0 and later on had 3D capabilities (actually more like 2.5 D). How difficult is it to develop a 3D extension for PyGamemaker? Just answer, do not implement

### Claude

**Short answer: hard — a multi-month undertaking, not an "extension" in the plugin sense, because of how this codebase is built, not because 3D math is inherently hard.**

**What GameMaker's 3D actually was.** GM6+'s `d3d_*` functions gave you a projection camera, a z-buffer, `draw_wall`/`draw_floor` (textured quads), a matrix transform stack, and simple lighting/fog — layered *on top of* the same 2D room/instance model. Instances still lived at 2D `(x,y)`; "3D" was really a drawing-mode you could turn on inside an event, not a different engine. That's exactly why it's called 2.5D: gameplay logic stayed 2D, only rendering went 3D.

**Why it's hard here specifically, in order of cost:**

1. **Triple implementation, by design.** This codebase's whole architecture (and everything enforced this session — behavior-preservation tests, the three-way parity discipline) treats desktop/pygame as the reference implementation and requires Kivy and HTML5 to match it action-for-action. I just checked: HTML5's canvas is `getContext('2d')` — no WebGL at all — and the Kivy exporter has zero OpenGL/3D usage; it only emits 2D `Line`/`Rectangle`/`Color` graphics instructions. So every 3D primitive would need three genuinely different implementations (pygame has no hardware 3D at all — you'd need to bolt in PyOpenGL alongside the existing `Surface`-blit pipeline; Kivy would need to drop down to its OpenGL layer, bypassing the whole existing 2D-canvas-instruction codegen; HTML5 would need to switch from Canvas2D to WebGL/three.js, a different rendering model with zero code reuse from the current `renderDrawCommands`). That's not "add a feature," it's "add three separate rendering backends and keep them in sync."

2. **No `z` axis anywhere in the data model.** Instances only carry `x`/`y` (`depth` exists but is a pure 2D draw-order sort key, not a spatial coordinate). Real 3D positioning, collision, and movement math (`move_towards`, `distance_to`, etc.) are all 2D vector code — introducing a genuine `z` would touch the instance model, the action executor, and every geometry-consuming action across all three runtimes.

3. **No IDE authoring surface for it.** The room editor is a flat pixel-grid canvas. Placing/orienting 3D geometry, defining a camera, or previewing perspective has no UI home today — that's its own substantial IDE project on top of the runtime work.

4. **This project's stated stance cuts against it.** `CLAUDE.md`'s "stability over features" position and the rc.11 "stop lying to users" cleanup (which specifically tore out half-working features rather than ship broken UI) both argue against a feature this large unless it's fully committed to across all three targets — a half-implemented 3D mode (works on desktop, no-ops on HTML5) is precisely the anti-pattern this codebase has been actively removing.

**What would actually be tractable** is not real 3D but a narrower "fake 2.5D" — e.g. a Doom-style textured floor/ceiling raycast or an isometric height-offset — implemented as **new draw-queue command types** (same mechanism `draw_rectangle`/`draw_health_bar`/etc. already use) computed in pure Python/JS with no `z` axis, no camera object, no OpenGL/WebGL dependency. That's a real extension-sized task (roughly comparable to this session's `draw_*` codegen work), not a rewrite — but it's also not "GameMaker's 3D," just a visual effect wearing its clothes. If you want that scoped out, it's a very different (and much smaller) conversation than "add `d3d_*` parity."
