## What I found before planning

Views/cameras are **already implemented on the desktop pygame runtime** — a genuinely complete GameMaker-style system: 8 views per room, border-based camera-follow with per-axis speed limiting, clamping to room bounds (`GameRunner.update_views()`, `runtime/game_runner.py:1247-1631`). `enable_views`/`set_view` action handlers exist and work.

But it's marked **"incomplete"** in `TODO.md` for two concrete reasons I confirmed:
1. **No IDE UI exposure** — `enable_views`/`set_view` have no entry in `events/action_types.py`, so there's no action-editor dialog or Blockly block for them. Usable today only by hand-authoring the JSON action dict or driving it from `execute_code` — exactly the pattern match3 already established.
2. **Zero export support** — `grep -rl "views_enabled\|view_offset" export/` returns **nothing**. Neither the Kivy exporter nor the Web/HTML5 exporter has any concept of a camera. A views-based game would export to Kivy/Web and either ignore the camera (render everything at absolute room coordinates, breaking the scroll) or worse.

That second gap is the same shape as the sound-queue/sprite-fallback gaps I closed for match3_2 this session — except bigger: sound was an *additive* queue bolted onto existing draw/sound paths. Views affect the **rendering transform of every single draw call**, so Kivy/Web support means threading a view offset through each exporter's whole render path, not adding one primitive.

## Proposed plan

**Given 68% of this session is already used, I'm proposing this as a plan to execute in a future session, not now.** Here's the shape I'd suggest:

**Phase 1 — engine work (do first, it's the actual risk):**
- Add view-offset-aware rendering to the Kivy exporter (base_object.py's render + draw-queue, scene camera transform) and to `engine.js` (canvas translate before drawing instances/tiles)
- Register `enable_views`/`set_view` in `action_types.py` so they're at least usable as structured actions (UI polish can stay deferred, but the metadata unblocks non-`execute_code` samples)
- Verify round-trip on a minimal synthetic room across all 3 targets before touching sample content

**Phase 2 — sample progression** (mirroring the maze_1→3 / match3_1→3 pattern, 2 samples rather than 3 given scope):
- **views_1** — single view, single large room (e.g. 2400×800 vs an 800×600 window), camera follows the player with GameMaker-style border scrolling. Minimal — proves the core mechanic.
- **views_2** — adds a second, smaller fixed-position view (e.g. a minimap in the corner) showing the same room at a different viewport, demonstrating multi-view.
