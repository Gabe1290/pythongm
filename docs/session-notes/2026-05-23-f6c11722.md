## [2026-05-23] User

> In what format are the sample games stored?

### Claude

**Storage format: native pygm2 project folder (not GMK).** Each sample is a directory of plain JSON files plus binary asset blobs (images, sounds, fonts). Concretely:

```
samples/maze_1/                  ← one project = one folder
├── project.json                 ← root manifest: name, version, settings, asset index
├── sprites/
│   ├── spr_person.png           ← raw image asset on disk
│   ├── spr_goal.png
│   └── spr_wall.png
├── objects/
│   ├── obj_person.json          ← one JSON file per object (events, behaviour)
│   ├── obj_goal.json
│   └── obj_wall.json
├── rooms/
│   ├── room0.json               ← one JSON file per room (instances, size, etc.)
│   └── room1.json
├── sounds/                      ← raw .wav / .mp3 / .ogg, indexed in project.json
├── backgrounds/                 ← raw image files
├── fonts/                       ← TTF / OTF
├── scripts/                     ← Python source files for project-level scripts
├── data/                        ← arbitrary user data
└── thumbnails/                  ← auto-generated preview images (cache; regen on save)
```

**Schema at a glance:**

| Layer | Format | Diffable? |
|---|---|---|
| `project.json` | JSON object: `{name, version, created, modified, settings, assets: {sprites, sounds, objects, rooms, …}}` | Yes — readable diffs |
| Per-asset metadata (events, room layout, etc.) | JSON files under `<category>/<name>.json` | Yes |
| Asset binaries (images, sounds, fonts) | Native formats (PNG, WAV/OGG, TTF) | No (binary), but easy to swap |

**Why this format:**

- **Loads directly via `load_project(folder)`** — no import step, no conversion at click-time.
- **Diffable in git** — the JSON layer makes it possible to review changes to events / rooms / settings as readable diffs. Binary assets are still binary, but the structural data isn't opaque.
- **Same format the IDE uses for any project the user creates** — `samples/maze_1/` is structurally indistinguishable from a project a teacher might create with File → New Project. That's why copy-on-open works: `shutil.copytree` clones the folder, the IDE opens the clone via the standard `load_project` path, no special-casing.

**What changed in rc.12:** before commit `f8a0eb7`, the samples shipped as opaque GMK files (binary, GameMaker 8.x format) and required an importer to run at click time. That's gone — now they ship pre-imported as the native folder format above. The `.gmk` originals are documented in `samples/README.md` as regeneration sources but aren't shipped in the repo.

If you want to inspect an asset directly: `samples/treasure/scripts/<name>.json` (script source + metadata), `samples/treasure/objects/<name>.json` (event definitions), `samples/treasure/rooms/<name>.json` (instance placements) — all open in any text editor.
