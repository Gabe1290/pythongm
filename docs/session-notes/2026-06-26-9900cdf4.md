All three are in `requirements.txt`, and `Pillow` is already a core dependency (it's needed just to run the IDE), so in practice the user usually only needs to add **kivy** and **pyinstaller**. Here's the practical picture.

## The key point: admin rights aren't needed for any of these

Admin is only required for *system-wide* installs (`sudo pip install`, or writing into the system Python). Every option below installs into the **user's own home directory**, which never needs elevation. Kivy, PyInstaller and Pillow all ship **pre-built binary wheels** for Python ≤3.13, so `pip install` just downloads them — no C compiler, no SDK, no admin.

## The one rule that matters

The exporter checks `import kivy` and runs PyInstaller using **the same Python interpreter that launched the IDE** (`sys.executable`). So the packages must land in *that* environment — not just "somewhere on the machine." This makes the method you choose depend on how the IDE is started:

**If the IDE runs from a virtualenv (the project's documented setup):**
```bash
# activate the SAME venv the IDE uses, then:
pip install -r requirements.txt        # or: pip install kivy pyinstaller pillow
```
Everything lands in the project's `venv/` folder, entirely in the user's space. No admin. (Note: `--user` is *ignored* inside a venv, so just plain `pip install` here.)

**If the IDE runs from a plain system Python the user can't write to:**
```bash
python -m pip install --user kivy pyinstaller pillow
```
This installs to `~/.local/...` (Linux/macOS) or `%APPDATA%\Python\...` (Windows) — no admin, and that user-site is automatically visible to the same interpreter.

## Caveats for genuinely locked-down (school) machines

1. **Python itself must already be present.** If the user is running the IDE *from source*, Python + pip are already there, so this is fine. If they only have a frozen IDE build with no Python, they can't pip-install at all.
2. **Firewall blocking PyPI.** If outbound access to pypi.org is blocked, do an offline bundle: on an unrestricted machine run `pip download kivy pyinstaller pillow -d wheels/`, copy the folder over, then `pip install --no-index --find-links wheels/ kivy pyinstaller pillow`.
3. **AppLocker / SmartScreen.** Some school machines block *running* freshly-built `.exe` files. That affects launching a PyInstaller export, not building it — a separate policy issue.

## My actual recommendation for this educational context

If your users are on locked-down school machines, the desktop-executable path is the *highest-friction* way to share a game — it needs the full Kivy + PyInstaller toolchain on the author's machine. Two lighter routes:

- **HTML5 export** needs **zero** install on the author's machine and the result runs in any browser — by far the lowest-friction "share my game" option for schools.
- **A built `.exe`/`.app` needs nothing installed on the *recipient's* machine.** Only the one person *building* the export needs the toolchain — students who just *play* the game don't.

So in a classroom, one prepared machine (or a teacher's) can build executables, while everyone else uses HTML5 or just runs the finished file.

> Just answer the following question, do not implemenr:
> When compiling the IDE, would it be possible to have a seperate executable containing kivy and pyinstaller that way we have two downloads for each platform: one without these libraries and one with eveything?

### Claude

Short version: **partly.** Bundling Kivy is easy; bundling PyInstaller in a way that actually *works* is the catch — and it changes what the "full" download can be.

## Why the two halves behave differently

**Kivy** is just a Python library. When you freeze the IDE with PyInstaller, you can absolutely include Kivy in the "full" build — the IDE's `import kivy` check would then pass. No problem.

**PyInstaller** is the blocker. There's a hard limitation: **you can't reliably run PyInstaller from inside a PyInstaller-frozen application.** PyInstaller works by spinning up a *real* Python interpreter, importing/analyzing the target script's modules, and bundling them together with that interpreter. Inside a frozen IDE:

- `sys.executable` points at the frozen IDE itself, not a Python interpreter — and the IDE's export code shells out via `sys.executable` to run the build.
- A frozen app doesn't carry the usable Python build scaffolding (a spawnable interpreter, the toolchain pieces) that PyInstaller needs.

So even if you stuffed the `pyinstaller` package into the bundle, the build step would have nothing real to drive it. The "full" exe couldn't actually produce game executables. (Note: this is about PyInstaller's nature, not Kivy vs pygame — switching the export runtime wouldn't dodge it.)

## So what *can* the "full" download be?

The workable shape of your idea isn't "one extra compiled .exe" — it's **the IDE shipped alongside a bundled, real (portable/embeddable) Python environment that already has kivy + pyinstaller + pillow installed**, with the IDE shelling out to *that* Python instead of to itself. Concretely per platform:

- **Windows** — embeddable-Python zip with the libs pip-installed, plus a launcher.
- **macOS** — a `.app` embedding a real `Python.framework` with the libs (codesigning/notarization adds friction).
- **Linux** — an AppImage or tarball carrying a portable Python.

That delivers exactly the goal — **everything included, no admin, no internet** — it's just a larger archive/installer (Kivy + PyInstaller + Python is easily a few hundred MB), not a single monolithic binary.

## Is the split worth it?

Yes, and it's a sensible distribution strategy:

- **Lite** (small, frozen IDE) still does almost everything — edit, Test Game, and **HTML5 export, which needs none of these libraries**. Good default for most users and for locked-down school machines.
- **Full** (IDE + bundled Python toolchain) is the "I want to build native .exe/.app files offline with no admin rights" download.

And remember the asymmetry that makes the lite build viable for classrooms: only the machine *building* a native export needs the toolchain — a finished `.exe`/`.app` runs on any machine with nothing installed.

One honest caveat: the "full" build roughly doubles your release-engineering work (you're now maintaining a packaged Python environment per platform, and on macOS dealing with signing a nested interpreter), so it's worth confirming there's real demand for offline native builds versus just steering those users to HTML5 or a one-time `pip install --user`.

## Root cause

The IDE runtime evaluates conditions like `vspeed > 0 and y < other.y+8` through `_eval_bool_expression`, whose `eval` namespace is **seeded with the instance's attributes** — so bare `vspeed`, `y`, `x`, custom vars all resolve. The Kivy exporter copied the author expression *verbatim* into the generated `if`, where those names are unbound → `NameError` every frame. (In the IDE this same class of error is silently swallowed; the export prints it loudly, which is why you saw the wall of tracebacks.)

## The fix

The generator now rewrites bare instance/custom names to `self.<name>` via an AST transform ([code_generator.py](export/Kivy/code_generator.py)), applied at the three raw-expression sites: `test_expression`, `if_condition`, and `while`.

| Author wrote | Now exports as |
|---|---|
| `vspeed > 0 and y < other.y+8` | `-self.vspeed > 0 and self.y < other.y + 8` |
| `my_counter >= 3` (custom var) | `self.my_counter >= 3` |
| `self.vspeed > 0` (already qualified) | unchanged |
| `other.vspeed < 0` | unchanged |
| `abs(hspeed) > 4` | `abs(self.hspeed) > 4` |
| `lives <= 0` (game global) | left bare |

`vspeed` is emitted in GameMaker sign (`-(self.vspeed)`) to stay consistent with the existing `test_variable` conditional, and malformed expressions pass through untouched so codegen never crashes.
