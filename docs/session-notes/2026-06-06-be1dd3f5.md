## The "Persistent" checkbox

It's an **object property** (in the Object Editor's properties panel, alongside *Visible* and *Solid*). Its tooltip says it plainly: **"Object persists between rooms."**

### What it does

Normally, when the game changes from one room to another, **every instance is destroyed** and the new room is built fresh from its own layout. Marking an object **Persistent** changes that for its instances: when you leave a room, a persistent instance is **carried over into the next room** instead of being destroyed.

You'd typically use it for things that should survive across rooms — the player character, a score/HUD controller, an inventory manager, background music controller, etc.

### How pygm2 actually implements it

Looking at [game_runner.py](runtime/game_runner.py#L4105-L4171), during a room change:

1. Before leaving, it collects all instances whose object has `persistent: True` ([line 4112](runtime/game_runner.py#L4112)).
2. In the new room, it **removes the room's default (non-persistent) copy** of that same object type, so the carried-over instance replaces it rather than duplicating ([lines 4127-4131](runtime/game_runner.py#L4127-L4131)).
3. The carried instance keeps its state, but pygm2 **refreshes its object_data** from the project and **resets hspeed/vspeed to 0** ([lines 4133-4143](runtime/game_runner.py#L4133-L4143)) — this is a deliberate fix to stop momentum carrying over (e.g., the player bouncing back onto a door).
4. The persistent instance does **not** re-run its `create` event in the new room — only genuinely new instances do ([lines 4167-4171](runtime/game_runner.py#L4167-L4171)). Its create event runs once, when first spawned.

### One subtlety worth knowing

`persistent` is one of four **child-only** object properties (along with `sprite`, `visible`, `solid`) — see [object_inheritance notes](editors/object_editor/object_properties_panel.py). That means if an object has a **parent**, it does **not** inherit `persistent` from that parent; each object's persistence comes from its own checkbox (or defaults to off).

### Note: rooms have a separate "persistent" flag too

Don't confuse the object checkbox with **room persistence** (`room_data['persistent']`, [game_runner.py:1165](runtime/game_runner.py#L1165)) — that's a different concept (preserving a whole room's state when you leave and return to it). Your question is most likely about the object checkbox, which is the one labeled "Persistent" in the Object Editor.

Crucially, unlike the room-*change* path, this code **does not look at the persistent flag at all.** It throws everything away and re-spawns the full original layout. So after a restart, every bonus comes back regardless of how any checkbox is set.

## Why Persistent is the wrong tool

The **Persistent** checkbox only carries an instance *that still exists* from one room into the **next** room (a room change, not a restart). Two reasons it can't keep bonuses deleted:

1. A collected bonus has already been **destroyed** — there's no instance left to "persist." Persistence carries living instances forward; it can't carry forward a deletion.
2. pygm2's restart path ignores persistence entirely and rebuilds from the authored layout anyway.

So checking Persistent on your bonuses would do nothing useful here (and if anything it's meant for the opposite kind of object — the player, a score/HUD controller, etc.).

## What would actually work

To make collected bonuses **stay gone across a restart**, you need to **remember that they were collected** in state that survives the rebuild, then have each bonus remove itself on spawn if it was already taken. The usual pattern:

1. Keep a **persistent controller object** (this is a legitimate use of the Persistent checkbox) or a **global variable** that records which bonuses have been collected — e.g. a counter, or a flag per bonus.
2. When the player collects a bonus, set that flag/global *before* destroying the bonus.
3. In each bonus's **create** (or **room start**) event, check the flag and `destroy_instance` itself immediately if it's already marked collected.

That way the restart re-spawns the bonus, but it deletes itself on the same frame, so the player never sees it.
