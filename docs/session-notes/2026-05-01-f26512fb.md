## What "GPU rendering" actually means here

pygame's default `Surface.blit` is **software rendering**: every pixel is composited by Python calling SDL functions that run on the CPU. The optimization would switch to **`pygame._sdl2.Renderer`** (or pygame-ce's modern API), which routes the same blit calls through SDL2's hardware-accelerated path.

That hardware path uses:

| Hardware | What gets called | Available on |
|---|---|---|
| **Integrated GPU** (Intel UHD, Iris, AMD Vega, Apple M-series) | DirectX 11/Direct3D, OpenGL, or Metal — through SDL | Every Windows 10/11 PC built in the last ~10 years, every Mac, most Linux desktops |
| **Discrete GPU** (NVIDIA / AMD card) | Same, just faster | High-end gaming PCs |

Decoded:
- **`mf_video_encoder_util.cc`** — a Chromium internal file (Media Foundation video encoder utility)
- **`Set output type failed`** — Chromium tried to negotiate a video-encoder output format with the Windows Media Foundation API
- **`0x80070057`** — Windows' generic `E_INVALIDARG` ("parameter is wrong")

## Why it fires

Chromium (which is what QtWebEngine wraps to display the Blockly page) probes your machine's video encoders at startup, trying to identify what's available for things like WebRTC video calls or HTML5 video element encoding. The probe asks Windows Media Foundation "can you encode H.264 with these parameters?" — your machine answers "no" — Chromium logs the failure to stderr and moves on to the next codec.

## Why it doesn't matter

PyGameMaker doesn't use any of this:
- No video calls in the IDE
- No HTML5 `<video>` element on the Blockly page
- No screen recording, no media capture

Chromium runs the probe regardless of whether anything will use it. The "failure" is just one combination not working; it falls back to others silently.

## Why we already partially handle it

Look at [main.py:230-237](main.py#L230-L237) — there's a Qt message-handler suppression list that already includes `"mf_video_encoder"` and `"Set output type failed"`. But that suppression only works on Qt's own logging channel; this particular line comes from Chromium writing directly to `stderr`, bypassing Qt.

## If you want it gone

```python
os.environ['QTWEBENGINE_CHROMIUM_FLAGS'] = (
    os.environ.get('QTWEBENGINE_CHROMIUM_FLAGS', '')
    + ' --disable-features=MediaFoundationVideoEncoder'
).strip()
```
